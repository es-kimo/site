# 시작하며

최근 gpt와 커리어 고민 상담을 했다. 내 질문은 `be를 공부한다면, fe 개발자로서 어떤 점을 집중해서 공부해야할까?`

> 이 질문은 아주 정확한 타이밍에 나왔어요.
> “BE를 할 줄 아는 FE”와 “BE 관점으로 설계하는 FE”는 완전히 다릅니다.

결론부터 말하면,

❌ “백엔드도 할 줄 아는 FE”가 아니라
✅ **“프론트엔드의 경계를 스스로 정의할 수 있는 FE”**가 되는 게 목표입니다.

아래를 우선순위 + 이유 + 구체 학습 포인트로 정리해볼게요.

⸻

0️⃣ BE를 왜 공부하는가? (목적 정렬)

FE 개발자가 BE를 공부하는 목적은 이겁니다.
• API를 더 잘 쓰려고
• 서버랑 싸우지 않으려고
• FE/BE 경계를 내가 설계하려고
• “이건 서버에서 해야 합니다”를 논리적으로 말하려고

👉 풀스택이 목표가 아니라 “설계 권한”이 목표

⸻

1️⃣ 1순위 — HTTP & 네트워크 (최우선)

이걸 모르면 FE는 항상 BE 뒤에 끌려다닙니다.

반드시 알아야 할 것
• HTTP 메서드의 의미 (GET/POST/PUT/PATCH/DELETE)
• 상태 코드의 의미
• Idempotency (중복 요청 안전성)
• Cache-Control / ETag / If-None-Match
• 쿠키 vs 로컬스토리지 vs 세션
• CORS가 왜 생겼는지
• 브라우저가 요청을 언제 막는지

FE 관점 포인트
• “이건 서버 캐시로 해결해야 한다”
• “이건 클라이언트에서 재요청하면 안 된다”
• “이 API는 설계가 잘못됐다”를 말할 수 있어야 함

📌 HTTP를 아는 FE는 이미 반쯤 시니어

⸻

2️⃣ 2순위 — API 설계 관점 (CRUD 말고)

API를 “쓰는 사람”에서 “설계 평가자”로 바뀌는 지점

집중 포인트
• 리소스 중심 REST 설계
• URL에 동사가 왜 나쁜지
• pagination / filtering / sorting
• 에러 설계 (에러 코드, 메시지, 필드 에러)
• 버전 전략 (v1, v2 vs 필드 확장)
• GraphQL을 쓰는 이유 / 안 쓰는 이유

FE에게 중요한 이유
• API shape가 UI 구조를 결정
• 나쁜 API는 FE를 복잡하게 만듦
• 좋은 API는 상태 관리가 사라짐

👉 “이 API는 프론트 상태를 줄여준다/늘린다” 판단 가능

⸻

3️⃣ 3순위 — 서버에서 ‘상태’가 어떻게 관리되는가

FE는 상태 지옥에 빠지기 쉽습니다.
해결책의 절반은 서버 이해에 있어요.

꼭 봐야 할 개념
• Stateless vs Stateful
• 인증 상태 (Session / JWT / Refresh Token)
• 서버 캐시 (Redis 등)
• 트랜잭션 개념
• 동시성 문제 (중복 요청, race condition)

FE에 바로 연결되는 질문
• 로그인 상태는 누가 기억해야 하는가?
• 새로고침 시 상태가 날아가면 정상인가?
• optimistic update가 가능한가?
• 중복 클릭 방지는 어디서 해야 하는가?

⸻

4️⃣ 4순위 — DB는 ‘모델링 관점’만 가져가라

FE가 SQL 고수가 될 필요는 없습니다.
데이터 구조를 이해하는 것이 목적입니다.

핵심만 가져가기
• 테이블 구조가 UI에 미치는 영향
• 1:N / N:M 관계
• soft delete vs hard delete
• pagination 비용
• “이건 JOIN 지옥이겠구나” 감각

👉 이걸 알면:
• API 요청 수 줄이는 설계 가능
• UI에서 비정상적인 데이터를 빨리 감지

⸻

5️⃣ 5순위 — 에러와 장애를 기준으로 생각하기

주니어와 시니어를 가르는 기준 중 하나

BE 관점 질문을 FE가 던질 수 있어야 함
• 이 요청이 실패하면?
• 부분 성공이면?
• 타임아웃이면?
• 재시도해도 되는 요청인가?
• 중복 요청 시 데이터 깨지나?

👉 “에러가 났을 때 UX까지 포함해서 설계하는 FE”

⸻

6️⃣ FE가 굳이 깊게 안 파도 되는 BE 영역

❌ 마이크로서비스 아키텍처 깊이
❌ 인프라 튜닝
❌ DB 쿼리 최적화 장인 레벨
❌ 특정 언어(Spring, Nest)의 내부 구현 디테일

👉 이건 전업 BE 트랙입니다.

⸻

7️⃣ 추천 학습 방식 (중요)

❌ 흔한 실패
• BE 강의 그대로 따라가기
• CRUD 서버 만들기만 반복
• “풀스택 느낌”에 만족

✅ 좋은 방식
• FE 문제에서 출발
• “이 상태를 FE에서 들고 가는 게 맞나?”
• “이건 API를 바꾸는 게 낫겠다”
• “이건 서버 책임이다”

📌 서버를 FE 문제 해결 도구로 쓰는 감각

⸻

8️⃣ 한 문장 요약

FE 개발자가 BE를 공부한다는 건
서버를 잘 만들기 위함이 아니라
프론트엔드의 경계를 스스로 정하기 위함이다.

최근에는 파일 zip의 책임이 서버에 있는지 클라이언트에 있는지 논의했었다.

물론이야!
아래는 1–5번 감각 (HTTP, API 설계, 서버 상태, 데이터 구조, 장애 감각) 을 체계적으로 키우는 데 정말 도움 되는 책들이야 — 프론트엔드/백엔드를 넘나들며 현업 감각을 단단히 만들어줄 추천 리스트야.

⸻

📚 1) HTTP & 네트워크 감각

🔹 “HTTP: The Definitive Guide” — David Gourley & Brian Totty
• HTTP 프로토콜을 가장 깊고 넓게 다루는 책.
• HTTP 메서드, 상태 코드, 캐싱, 연결 관리, 보안까지 상세.
👉 네트워크 기반 감각의 기본기 뿌리

⸻

🔹 “High Performance Browser Networking” — Ilya Grigorik
• 브라우저 관점에서 HTTP/2, TCP/UDP, 캐싱, 렌더링 최적화
👉 FE에서 HTTP를 단순 요청이 아니라 성능 관점으로 볼 수 있게 해줌

⸻

📚 2) API 설계 감각

🔹 “Designing Web APIs” — Brenda Jin, Saurabh Sahni, Amir Shevat
• RESTful API의 설계 원칙, HTTP 메서드 사용, 에러 핸들링, 버전관리 등
👉 API가 어떻게 설계돼야 FE를 편하게 해주는지 체계적으로 이해 가능

⸻

🔹 “A Practical Approach to API Design” (Leanpub)
• API 설계의 취지, 계약 기반 설계(API-First), 스펙 중심 개발 등
👉 FE/백엔드 간 Contract Thinking(계약 우선 설계) 이해에 강함 ￼

⸻

🔹 “웹 API 설계 원칙” — 제임스 히긴보텀
• REST API 설계 원칙을 실용적으로 다루는 한글 서적
👉 API 설계의 이유와 목적을 이해하는 데 도움 됨 ￼

⸻

📚 3) 서버 상태 & 백엔드 실무 감각

🔹 “Designing Data-Intensive Applications” — Martin Kleppmann
• 데이터 흐름, 상태 일관성, 장애 대응, 분산 시스템 개념까지
👉 “서버가 상태를 어떻게 책임지는지” 체계적 통찰 제공 ￼

⸻

🔹 “주니어 백엔드 개발자가 반드시 알아야 할 실무 지식” — 한빛미디어
• HTTP, 장애 대비, 동시성, DB 성능 등 실무 문제 해결 중심
👉 실제 서비스에서 발생하는 다양한 오류/상태 케이스를 배울 수 있어 현실적 ￼

⸻

📚 4) 시스템 & 데이터 구조 감각

🔹 “요즘 개발자를 위한 시스템 설계 수업” — 디렌드라 신하
• 시스템 설계 전반, API 설계, 장애 허용성, 캐시, 큐, 분산 구조까지
👉 FE가 시스템을 크게 보는 감각을 키울 수 있는 책 ￼

⸻

🔹 “Clean Architecture” / “Clean Code” — Robert C. Martin
• 소프트웨어 구조와 코드 품질의 본질을 다루는 클래식
👉 API/데이터 구조가 FE 코드를 어떻게 지저분하게 또는 깔끔하게 만드는지 감각 상승

⸻

📚 5) 장애 감각 & 운영 관점

🔹 “Site Reliability Engineering” — Google SRE Team
• 장애 시나리오, 모니터링, 경보 전략 등 “운영” 관점의 성경
👉 단순 에러 처리 수준을 넘어 운영 사고까지 상상하는 감각 형성

⸻

🔹 “The Phoenix Project” / “The DevOps Handbook”
• 개발-운영의 현실적 협업, 장애 대응, 피드백 루프 등
👉 개발자가 장애를 ‘운영 차원’으로 생각하는 관점 강화 ￼

⸻

📌 실전 활용법 (책 + 루틴)

📍 HTTP 감각
• 책 → 예시 코드 → 실제 네트워크 탭 분석 루틴

📍 API 설계
• 책 → OpenAPI/Swagger로 스펙 직접 작성 + API 계약 리뷰

📍 서버 상태
• 책 → 데이터 모델을 UI에 맞게 재구조화하는 실습

📍 데이터 구조
• 시스템 설계 책으로 대규모 파이프라인 모델 그려보기

📍 장애 감각
• SRE 책으로 장애 대응 문서 만들고 케이스별 대응 설계
