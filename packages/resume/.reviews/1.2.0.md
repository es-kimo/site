#### 1. 고객지원 시스템 ⭐ Case Study 후보로 강력 추천

CTO가 후킹될 포인트:

- **RBAC 구현** — 조직관리 시스템과 연동이라는데, 구체적으로 권한 모델이 어떻게 생겼는지(Role 계층? Permission 기반?), FE에서 권한 체크를 어떻게 했는지(라우트 가드? 컴포넌트 레벨? HOC?).

> **Case Study 방향**: "4가지 사용자 맥락(고객 모바일/데스크톱, 직원 모바일/데스크톱)을 하나의 코드베이스에서 어떻게 분리/공유했는가" → 아키텍처 설계 역량을 보여주는 이야기

#### 2. 판매관리 시스템 ⭐ Case Study 후보

- **"클라이언트 단 zip 암호화 (윈도우/맥 지원)"** — 이건 매우 흥미로운 기술적 도전입니다. 브라우저에서 zip 암호화를 왜 클라이언트에서 한 건지(서버 부하 때문이라고 했지만), 어떤 라이브러리/API를 썼는지, 크로스 플랫폼 이슈는 뭐였는지.
- **"API 모킹 커버리지 90%"** — 왜 90%인지, 나머지 10%는 뭔지, 모킹 전략(MSW? 커스텀?)이 궁금합니다. 이건 팀 생산성에 직결되는 이야기라 CTO가 좋아합니다.

> **Case Study 방향**: "대형 폼 시스템을 제네릭으로 추상화한 설계 과정" 또는 "API 모킹 전략으로 팀 코드 리뷰 경험을 개선한 이야기"

## 진짜 깊이가 있는 것 (CTO가 "오?" 할 만한 것)

### 1. Ruler for Tablet — FSM 설계 ✅

**이건 진짜입니다.** 이유:

- "한 손가락 터치 하나로 5가지 동작을 구분해야 한다"는 제약 조건 자체가 비자명(non-trivial)합니다. 키보드+마우스 환경에서는 modifier key(Ctrl, Shift)로 쉽게 분기하지만, 터치에서는 **같은 입력으로 다른 의도를 추론**해야 합니다.
- FSM이라는 해결책이 문제에 정확히 맞습니다. 상태 폭발 없이 입력을 분류하려면 상태 기계가 거의 유일한 정답이고, 이걸 스스로 도출한 건 설계 판단력입니다.
- 다만 **현재 Case Study 서술이 이 깊이를 전혀 살리지 못하고 있습니다.** "옵셔널체이닝을 정확한 순서로 사용"이 execution의 한 항목인 건 솔직히 아깝습니다. 상태 전이 규칙, 엣지 케이스(예: 줌 중에 손가락을 떼면?), hit testing 로직이 빠져 있습니다.

### 2. 대량 Point 관리 — 데이터 구조 개선 → 렌더링 성능 ✅ (잠재력)

> "리액트가 key를 활용한 렌더링 성능을 유지할 수 있도록 불필요한 계산을 찾아 데이터 구조 개선"

이건 **제대로 풀어쓰면** 깊이가 있습니다. "React의 reconciliation에서 key가 어떻게 동작하는지 이해하고, 데이터 구조를 그에 맞게 바꿨다"는 건 프레임워크 내부 동작을 이해한다는 증거입니다. 하지만 현재는 한 줄짜리 bullet으로만 있어서, **구체적으로 뭘 했는지 전혀 모릅니다.** 기존 구조가 뭐였고, 어떤 계산이 불필요했고, 어떻게 바꿨고, 성능이 얼마나 개선됐는지 — 이 중 하나라도 있어야 깊이가 보입니다.

### 3. 클라이언트 단 zip 암호화 (윈도우/맥 지원) ✅ (조건부)

**"왜 클라이언트에서?"**라는 질문에 납득할 답이 있다면 깊이가 있습니다. 브라우저에서 대용량 파일을 암호화 zip으로 만드는 건:

- Web Streams API / ReadableStream으로 메모리 제어
- 크로스 플랫폼 암호화 호환성 (Windows 기본 ZIP이 AES를 안 쓰는 등)
- Worker thread 사용 여부

이 중 실제로 문제를 겪고 해결한 게 있다면 좋은 소재입니다. 단순히 `@zip.js/zip.js` 갖다 썼으면 깊이가 아닙니다.

## 깊이가 있어 보이지만 실제로는 약한 것

### 4. "아키텍처 설계에 주도적으로 기여" ❌

고객지원 시스템, 판매관리 시스템 둘 다 이 문구가 있는데, **이건 가장 위험한 표현**입니다. CTO는 이걸 보면 즉시 "구체적으로 뭘 설계했는데?"라고 묻습니다. 현재 resume에는 그 답이 없습니다. 이전 대화에서 분석했듯, 실제 구조(모노레포 2앱 + shared)는 합리적이지만 개별 구현 난이도는 높지 않습니다.

### 5. "Form Field 컴포넌트 8종을 제네릭으로 Type-safe하게" ❌

react-hook-form 위에 래핑한 건 팀에서는 의미 있지만, 기술적 깊이라고 부르기 어렵습니다. `Controller<TFieldValues>`에 제네릭 넘기는 건 공식 문서 패턴입니다. `Path<T>` 같은 recursive type을 직접 만들었다면 다르지만, react-hook-form이 이미 제공하는 걸 쓴 거라면 "잘 활용했다" 수준입니다.

### 6. "API 모킹 커버리지 90%" ❌

깊이보다는 **성실함**의 영역입니다. MSW 핸들러를 90% 커버리지로 유지하는 건 좋은 습관이지만, 기술적 도전이 아닙니다. "왜 90%인지, 나머지 10%는 구조적으로 모킹이 불가능했는지"를 설명할 수 있으면 조금 나아지지만, Case Study 소재로는 약합니다.

### 7. "RBAC 구현" ❌

이전에 코드를 봤습니다. 문자열 비교로 `isAdmin`, `isManagerOrAbove` 불리언을 만드는 수준입니다. Permission 기반 동적 권한 모델도 아니고, 라우트 가드도 아닙니다. FE RBAC로서는 표준적인 구현이고, 깊이가 있다고 말하기 어렵습니다.
